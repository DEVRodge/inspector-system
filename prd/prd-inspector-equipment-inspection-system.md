# 产品需求文档：光伏厂区设备巡检数字化系统

**文档版本：** V1.0  
**编制日期：** 2025-09-12  
**来源文档：** 厂区设备巡检数字化建设方案 v1.0.1

---

## 1. 引言 / 概述

### 1.1 背景与问题

厂区逆变器巡检过程中存在以下痛点：

- **纸质记录效率低**：手工填写巡检表，易出错、难汇总
- **执行过程不可追溯**：无法确认巡检人员是否到达现场、何时完成
- **异常闭环不及时**：问题发现后缺乏标准化处置流程，整改跟踪困难
- **统计分析困难**：数据分散，难以形成决策依据

### 1.2 目标与价值

建设一套覆盖**现场巡检**与**后台管理**的数字化巡检系统，实现：

- 每台逆变器**唯一编码**、**唯一二维码**、**唯一巡检档案**
- 巡检人员通过安卓终端扫码执行巡检，实时提交结果、照片和异常
- 管理人员通过 PC 端统一查看任务执行、异常整改及设备运行趋势
- 形成完整的**巡检管理闭环**

### 1.3 系统组成

| 组件 | 说明 |
|------|------|
| PC 管理端 | 设备台账、任务管理、异常处理、统计报表、系统配置 |
| 安卓巡检端 | 巡检任务、扫码作业、表单填写、异常上报、消息通知 |
| 后台服务平台 | 数据存储、接口服务、权限控制、消息推送 |
| 二维码标识体系 | 设备唯一标识、现场扫码入口 |

## 2. 建设目标（Goals）

| 序号 | 目标 | 说明 |
|------|------|------|
| G1 | 建立设备电子台账 | 逆变器基础资料集中化管理 |
| G2 | 建立二维码识别机制 | 设备身份唯一化，扫码巡检标准化 |
| G3 | 建立移动巡检作业模式 | 巡检过程电子化、留痕化、可追溯 |
| G4 | 建立异常闭环机制 | 问题发现、派发、处理、验证全过程管理 |
| G5 | 建立多维统计分析 | 为设备管理、班组考核和管理决策提供依据 |

### 4.3 核心业务规则

| 规则 ID | 规则描述 |
|---------|----------|
| BR-01 | 每台逆变器必须建立唯一设备档案，并与唯一二维码绑定 |
| BR-02 | 二维码不得重复复用 |
| BR-03 | 巡检任务支持自动生成与人工新增两种方式，以计划周期为主 |
| BR-04 | 巡检人员原则上必须通过现场扫码方式进入巡检页面，系统应尽量限制脱离现场的手工补录行为 |
| BR-05 | 巡检记录提交后应保留原始数据，不得直接物理删除，应通过审计方式留痕管理 |
| BR-06 | 当巡检结果中存在异常项时，系统应自动生成异常记录并进入处置流程 |
| BR-07 | 系统应对逾期未完成任务、超期未闭环异常进行提醒并纳入统计考核 |
| BR-08 | 同一设备在同一巡检周期内重复提交时，**覆盖原来的巡检记录**（以最新提交为准） |

## 5. 非功能需求（Non-Functional Requirements）

### 5.1 易用性

| 需求 ID | 需求描述 |
|---------|----------|
| NFR-01 | 安卓客户端界面布局清晰，按钮触控范围适中 |
| NFR-02 | 扫码后进入巡检表单的时间应尽量缩短，减少现场等待 |
| NFR-03 | 表单录入过程应尽量减少不必要输入，提高单次巡检完成效率 |

### 5.2 性能

| 需求 ID | 需求描述 |
|---------|----------|
| NFR-04 | 普通业务页面响应时间不超过 3 秒 |
| NFR-05 | 扫码后设备信息和表单加载时间不超过 2 秒 |
| NFR-06 | 正常网络环境下，巡检记录提交时间不超过 5 秒 |

### 5.3 稳定性与安全

| 需求 ID | 需求描述 |
|---------|----------|
| NFR-07 | 关键业务接口具备失败重试机制 |
| NFR-08 | 支持弱网或断网场景下的本地暂存与恢复提交 |
| NFR-09 | 采用账号认证与角色授权机制，通过 HTTPS 保障数据传输安全 |
| NFR-10 | 关键操作、异常处理、数据提交等行为应保留审计日志 |

### 5.4 可扩展性

| 需求 ID | 需求描述 |
|---------|----------|
| NFR-11 | 后续可扩展更多设备类型及巡检模板（配电室、开关柜、泵房、电机等） |
| NFR-12 | 后续可扩展定位打卡、电子签名、NFC、蓝牙识别等能力 |
| NFR-13 | 后续可对接企业 ERP、EAM、MES、统一消息平台或大屏可视化系统 |

## 7. 设计考虑（Design Considerations）

### 7.1 PC 管理端

- 采用表格、筛选、分页等常见管理后台模式
- 统计报表需支持图表展示（柱状图、折线图、饼图等）
- 异常管理需支持流程状态可视化（待指派、处理中、待整改、已闭环）

### 7.2 安卓巡检客户端

- 移动端优先，适配现场单手操作
- 扫码入口需置于首页显著位置
- 表单填写需支持快速切换（单选、数值、拍照）
- 弱网场景需有明确提示和重试按钮

### 7.3 二维码

- 二维码尺寸需满足现场打印张贴后清晰可扫
- 建议包含设备编码、设备名称等可读信息

## 8. 技术考虑（Technical Considerations）

### 8.1 架构

- PC 管理端：Web 应用，支持主流浏览器
- 安卓巡检端：**原生 Android 开发**（需支持摄像头扫码）
- 后台：RESTful API，支持 HTTPS 与 Token 认证

### 8.2 数据模型

| 数据类别 | 主要字段 |
|----------|----------|
| 设备主数据 | 设备 ID、设备编码、设备名称、二维码编号、设备类型、型号规格、所属区域、安装地点、责任班组、责任人、投运日期、运行状态 |
| 巡检任务 | 任务 ID、任务名称、巡检类型、计划开始/结束时间、执行人员、关联设备、任务状态、超时标识 |
| 巡检记录 | 记录 ID、任务 ID、设备 ID、扫码时间、巡检时间、提交时间、巡检人、巡检结果、异常标识、备注、附件地址 |
| 异常处理 | 异常 ID、来源记录、异常等级、异常描述、上报时间、上报人、指派人、处理人、处理措施、整改结果、闭环时间、闭环状态 |

### 8.3 部署需求

| 资源 | 规格 |
|------|------|
| 服务器 | 单路或以上规格 |
| 管理电脑 | 一套 |
| 巡检终端 | 三防型 PAD 终端一用一备 |
| 网络 | 具有公网 IP 的互联网或覆盖良好的园区无线网络 |

| 指标 | 目标 | 测量方式 |
